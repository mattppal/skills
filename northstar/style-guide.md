# Writing Style Guide

## Voice

- Start with a short, impactful quote or observation
- Use numbers and precise data points to create contrast
- Build narrative through personal experience
- Transition from specific to general insights
- Write in a teaching voice that reveals discoveries
- Use technical terms when needed, but explain through metaphor
- Maintain a conversational yet authoritative tone
- Punchy and opinionated—take a stance
- Write with tasteful humor sparingly
- Communicate effectively without unnecessary words

## Structure
- Short paragraphs that communicate effectively (avoid being too terse)
- Strategic use of single-line paragraphs for emphasis
- Start sections with concrete examples
- Build to abstract concepts gradually
- Use clear section breaks with descriptive headers
- End sections with broader implications
- Include relevant visuals to reinforce key points
- Aim for rhythm and flow between sections
- Write complete sentences, not sentence fragments
- Balance brevity with clarity—don't sacrifice meaning for shortness

## Language

- Blend technical precision with everyday language
- Use numbers strategically to illustrate points
- When using technical terms, provide immediate real-world context
- Keep technical explanations grounded in practical examples
- Use questions to drive narrative forward
- Target 4th-5th grade reading level for explanations
- Use American English spelling
- Avoid jargon and cliches
- Avoid common patterns like "Complex product?" that feel formulaic
- Write like a dev walking through their solution—active voice, explanatory
- Balance high-level concepts with detailed enough language that practitioners can implement

### Word Choice Guidelines

- Precise numbers when they serve the narrative
- Technical terms balanced with accessible language
- Active verbs that convey movement or change
- Contractions for natural flow
- Simple alternatives for complex terms
- Industry-standard terminology when discussing technical concepts
- Use "Take [example]" instead of "Example:" to maintain sentence flow
- Avoid overused transitions—make each sentence earn its place

## Orwell's Writing Rules

Ask the following questions:

- What am I trying to say?
- What words will express it?
- What image or idiom will make it clearer?
- Is this image fresh enough to have an effect?
- Could I put it more shortly?
- Have I said anything that is avoidably ugly?

Use the following rules:

1. Never use a metaphor, simile, or other figure of speech which you are used to seeing in print
2. Never use a long word where a short one will do
3. If it is possible to cut a word out, always cut it out
4. Never use the passive where you can use the active
5. Never use a foreign phrase, a scientific word, or a jargon word if you can think of an everyday English equivalent
6. Break any of these rules sooner than say anything outright barbarous

## Grammar
- Use Oxford commas
- Place periods inside quotation marks
- One space after periods
- Em dashes without spaces—like this
- Ellipses sparingly... and only for trailing thoughts
- Active voice over passive
- Present tense when possible
- Personal pronouns (I, you, we) to create connection
- Consistent verb tense within paragraphs

## Formatting
- Use **bold** for key concepts and emphasis
- Employ markdown for links and references
- Include relevant images with clear alt text
- Short lists can be inline with commas
- Longer lists should use bullet points
- Use section breaks to create rhythm
- Strategic use of *italics* for emphasis
- Em dashes (—) for asides and emphasis
- Code blocks for technical snippets
- Consistent heading hierarchy (H1 > H2 > H3)

## Headlines
- Use sentence case for all headers (capitalize first word and proper nouns only)
- Clear and punchy headers
- Use questions or statements
- Avoid clickbait
- Keep headers under 60 characters
- Make headers descriptive but concise
- Ensure headers follow logical progression

Examples:
- ✓ "Why we chose React for our frontend"
- ✓ "Building apps with Next.js"
- ✓ "How to implement dark mode"
- ✗ "Why We Chose React For Our Frontend"
- ✗ "Building Apps With Next.js"

## Narrative Structure
- Start with a hook that captures attention
- Build tension through personal discovery
- Use questions to surface reader thoughts
- Create "aha moments" through examples
- Connect technical concepts to everyday life
- End with forward-looking implications
- Maintain consistent pacing throughout

## Metaphors and Analogies
- Use familiar objects to explain complex ideas
- Build extended metaphors throughout sections
- Connect technical concepts to physical experiences
- Use analogies that simplify rather than complicate
- Return to established metaphors for reinforcement
- Ensure metaphors are culturally inclusive

## Examples

### Strong Openings
```markdown
"42% battery remaining."
"I drove an EV for the first time this weekend and I noticed one thing:"
```

### Effective Transitions
```markdown
"The thing is, I *rarely* think about..."
"Two things stuck out to me: **precision** and **salience**."
```

### Powerful Metaphors
```markdown
"From a battery percentage to a simple indicator with a warning light."
"Scarcity is the enemy of creativity."
```

### Strong Closings
```markdown
"We're incredibly early in the days of AI billing and there's a wide margin to improve."
"I, for one, would like a little less precision in my life."
```

## Article Structure
1. Hook (striking observation or quote)
2. Personal context
3. Problem introduction
4. Key concepts through metaphor
5. Real-world examples
6. Technical insights
7. Broader implications
8. Forward-looking conclusion

## Visual Elements
- Use images to reinforce key concepts
- Place visuals after introducing their context
- Ensure images add value to the narrative
- Include descriptive alt text
- Break up long sections with visual elements
- Optimize images for web performance
- Use consistent image styling

## Technical Writing
- Start with the familiar before introducing technical concepts
- Use real-world analogies for complex ideas
- Balance precision with accessibility
- Include links to technical references when needed
- Explain jargon through everyday examples

### Review Checklist

- Sentences over 25 words
- Paragraphs over 4 sentences
- Unexplained technical terms
- Passive voice constructions
- Overly complex explanations
- Consistency in terminology
- Image accessibility
- Link functionality
- Sentence fragments that should be complete sentences
- Cliches and common patterns (e.g., "Complex product?")
- Content that's too terse—does it communicate clearly?
- Bullet points—are they parallel in structure?

## Voice Examples

### Too Formal (Avoid)
```markdown
"One must consider the implications of external validation in relation to personal growth metrics."
```

### Just Right
```markdown
"Looking inward is hard, because you find imperfections. If you're someone who cares a lot, those imperfections can be daunting."
```

### Too Casual (Avoid)

```markdown
"Like, it's super hard to look at yourself and stuff, ya know?"
```

### Too Terse (Avoid)

```markdown
"Writing is fast. Everything around writing is slow."
```

### Better (Complete Thought)

```markdown
"Writing requires thought and attention. That's the differentiated skill. Everything else—branches, navigation, PR setup—is undifferentiated work."
```

## Key Lessons

- **Write for the reader, not yourself.** Ask: "Does this communicate clearly to my audience?"
- **Complete sentences matter.** Sentence fragments work for emphasis, but overuse makes content feel choppy
- **Avoid formulaic patterns.** Questions like "Complex product?" feel like templates, not authentic writing
- **Balance is key.** Don't be so terse that you lose meaning, but don't be verbose either
- **Make bullet points parallel.** If one starts with a verb, they all should
- **Foreshadow, don't repeat.** Tease what's coming without giving away all the details
